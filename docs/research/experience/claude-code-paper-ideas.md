## 原始分享

以下内容由用户提供，保留原作者的主要措辞，仅进行段落与 Prompt 排版：

随便问 LLM“你有什么 Idea”的时代结束了，现在是 Agent 时代。

上一篇笔记竟然有五千多收藏，看来大家对 CC 提效科研很感兴趣，那我就再开源一些自己的工作流，方便大家了。AI 科研这种话题争议太大，我只能每个笔记都叠甲：一天想十几个 Idea 是个人实测，但真正 work 的概率小于 X 神抽卡。这篇笔记能带给你的是如何每次都能吃到保底。同学们不要幻想天上掉 CCF-A，踏踏实实做实验才是王道。任何工具都无法代替个人努力的重要性。

先给结论：Claude Code + 人类监督可以稳定产出大量的、可供验证的 Idea。

### 为什么直接和 LLM 讨论 Idea 行不通

- 现在水论文的太多，大模型并没有足够聪明到识别哪篇论文是真的好，哪篇论文是在包装。
- 看了半天发现论文开源的代码一坨翔，你不炸了吗？

### 找 Idea 并验证的工作流

确定你要做的方向。你不能让 CC 凭空想一个 Idea，你需要划定一个大致的范围。搜集相关论文时，不要依赖大模型，必须由人类专家自己把关参考论文。

作者推荐的论文：模块化的、偏应用的。

作者不推荐的论文：探索性质的、没有开源代码的。

#### Prompt 1

```text
在这个项目建立两个文件夹：source 和 papers。
source 内含开源代码，papers 存论文原文 PDF。
```

#### Prompt 2

```text
请仔细阅读《论文 A》原文，开源代码我已放在 XXX 路径，请你尝试复现。
复现完成后请撰写笔记，要包含该论文的核心优缺点。
注意要从代码出发，不要被论文的包装所影响。
笔记放在 analysis 文件夹。
```

重复以上步骤，直至所有论文复现完成。

这里我要提醒各位同学：CC 不能顺畅干完这些。这一流程的完成建立在你本人对该领域的详细了解，知道代码为什么要这么写，为什么数据要这么组织。你要有自己的科研 Taste，能大致看出一个设计的好坏。

接下来进行 Idea 的生产。

#### Prompt 3

```text
请阅读 analysis 中的所有笔记，思考有没有办法能在 XXX baseline 的基础上做得更好。
```

这时候 CC 可以给出一些可行方案，人类专家需要根据自己的科研素养进行判断哪个最有希望。

#### Prompt 4

```text
使用 XXX 方案。
不要更改原有代码，所有的更改都要自己编写新的代码。
```

注意这里一定不要改原有代码。这一步需要多次迭代、返工，千万不要把最初的代码搞没了。同时，最初的代码也起到一个锚定作用，让 CC 能够在一个“安全域”内进行探索。

原文标签：论文、Agent、大模型。

## 结构化拆解

## 原文核心观点

“随便问 LLM 你有什么 Idea”的阶段正在过去。作者认为，更有效的方式是把 Claude Code（下文简称 CC）作为能够阅读论文、运行代码、记录分析和迭代实验的 Agent，并由具备领域知识的人持续监督。

作者给出的结论是：

> Claude Code + 人类监督，可以相对稳定地产出大量可供验证的 Idea。

这里的关键词不是“自动产生好论文”，而是**可供验证**。作者也特别强调：一天产生多个 Idea 并不意味着它们大多有效，真正可行的概率仍然很低；工具不能替代研究者扎实地做实验。

## 为什么直接与 LLM 讨论 Idea 不够可靠

作者提出两个主要问题：

1. **模型难以可靠判断论文质量。** 论文数量庞大，表达包装和真实贡献并不总是一致；大模型未必能够识别哪些工作真正扎实。
2. **论文叙述不能替代代码验证。** 阅读论文后如果发现开源实现质量很差、难以运行或与论文表述不一致，基于论文文本形成的判断就可能失真。

因此，该流程把重点从“让模型凭空想 Idea”转向：

```text
人工确定方向
→ 人工筛选论文
→ Agent 从代码出发复现
→ 形成结构化分析
→ 基于可运行 baseline 生成方案
→ 人工判断并做实验
```

## 第一步：由人确定研究范围

不能让 Agent 在没有边界的情况下凭空提出方向。研究者需要先划定一个相对明确的研究范围，并亲自筛选参考论文。

### 作者推荐的论文特征

- 模块化；
- 偏应用；
- 有开源代码；
- 具备可运行、可比较的 baseline。

### 作者不推荐的论文特征

- 偏探索性质，难以建立清晰验证路径；
- 没有开源代码；
- 无法通过代码和实验核查关键结论。

> [!WARNING]
> 论文筛选不能完全交给大模型。参考工作的质量、相关性和研究价值仍需要领域研究者把关。

## 第二步：建立论文与代码工作区

作者建议先建立两个目录：

```text
project/
├── source/   # 论文对应的开源代码
└── papers/   # 论文 PDF 原文
```

### Prompt 1：创建资料目录

```text
在这个项目建立两个文件夹：source 和 papers。
source 内含开源代码，papers 存论文原文 PDF。
```

这一结构让 Agent 同时拥有论文叙述和代码实现两个信息源，为后续交叉验证提供基础。

## 第三步：逐篇复现并形成分析笔记

对每一篇论文，要求 Agent 阅读原文、检查代码并尝试复现。分析不能只复述论文摘要，而要从实际实现和运行结果出发。

### Prompt 2：复现论文并记录优缺点

```text
请仔细阅读《论文 A》原文，开源代码我已放在 XXX 路径，请你尝试复现。
复现完成后请撰写笔记，要包含该论文的核心优缺点。
注意要从代码出发，不要被论文的包装所影响。
笔记放在 analysis 文件夹。
```

对所有候选论文重复该步骤，形成类似结构：

```text
project/
├── papers/
├── source/
└── analysis/
    ├── paper-a.md
    ├── paper-b.md
    └── paper-c.md
```

每份分析至少应该回答：

- 代码能否顺利运行；
- 论文关键结论是否能复现；
- 数据是如何组织的；
- 模型或算法由哪些模块构成；
- 哪些设计是真正有效的；
- 哪些优势可能主要来自实验设置或表达包装；
- 存在哪些可以进一步验证的缺陷。

## 人类监督是流程成立的前提

作者提醒，Claude Code 无法独立且顺畅地完成全部工作。研究者本人必须：

- 熟悉该领域；
- 理解代码为什么这样实现；
- 理解数据为什么这样组织；
- 能够识别复现结果是否合理；
- 对设计质量有基本判断；
- 拥有自己的研究 Taste。

换句话说，Agent 放大的是研究者已有的判断能力，而不是凭空替代它。

## 第四步：基于全部分析生产候选 Idea

完成论文复现和结构化分析后，再让 Agent 综合阅读所有笔记，并围绕一个明确 baseline 提出改进方案。

### Prompt 3：从分析中提出改进方案

```text
请阅读 analysis 中的所有笔记，思考有没有办法能在 XXX baseline 的基础上做得更好。
```

此时产生的候选方案有更扎实的上下文：它们来自论文原文、代码实现、复现结果和多篇分析之间的比较，而不是一次孤立的聊天。

但候选方案仍需要人类根据研究素养判断：

- 是否有明确贡献；
- 是否只是已有模块的简单拼接；
- 是否存在可验证的假设；
- 实验成本是否可接受；
- 是否可能仅在特定数据集上有效；
- 哪一个方案最值得优先投入。

## 第五步：在安全域内实现和迭代

选择候选方案后，让 Agent 编写新的实现，但不要直接修改原始 baseline。

### Prompt 4：保留原始代码实现新方案

```text
使用 XXX 方案。
不要更改原有代码，所有的更改都要自己编写新的代码。
```

作者强调，保留原始代码有两个作用：

1. **防止实验迭代破坏最初可运行的版本。** Idea 验证通常需要多次返工，必须保留随时可回退和比较的基线。
2. **为 Agent 提供探索锚点。** 原始实现定义了一个相对稳定的“安全域”，新方案可以围绕它增量探索，而不是让代码结构在多轮修改中失控。

更稳妥的工程结构可以是：

```text
project/
├── baseline/           # 保持只读或仅做必要配置
├── experiments/
│   ├── idea-001/
│   ├── idea-002/
│   └── idea-003/
├── analysis/
└── results/
```

## 工作流摘要

```text
1. 人工确定研究方向与边界
2. 人工筛选模块化、偏应用、带开源代码的论文
3. 建立 papers/ 与 source/
4. Agent 逐篇阅读、运行、复现并写入 analysis/
5. 人类检查复现过程和分析质量
6. Agent 综合分析，围绕明确 baseline 生成候选方案
7. 人类选择最有希望且可验证的方案
8. 保留原始代码，在独立目录实现新方案
9. 多轮实验、失败分析与返工
10. 以实验结果决定是否继续，而不是以 Idea 数量判断产出
```

## 待本站验证的问题

在将该方法升级为“个人 SOP”前，还需要通过真实任务验证：

- CC 阅读论文 PDF 与代码时，如何防止遗漏关键实验条件；
- 什么样的复现完成标准才足够严格；
- 如何记录环境、依赖、数据版本与随机种子；
- 如何避免 Agent 在分析笔记中编造不存在的运行结果；
- 多篇论文的分析格式是否需要统一模板；
- 如何对大量候选 Idea 做去重、新颖性检索和优先级排序；
- baseline 只读约束应依赖目录规范、Git 分支、容器快照还是沙箱；
- 如何衡量该流程相对普通 LLM 对话的真实增益。

这些问题验证完成后，才适合将流程提炼到“个人 SOP”和“内化”板块。
